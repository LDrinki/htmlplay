## 1. Product Architecture

**Decision: Static-first, JSON-chunked, vanilla JS modules.**

No React, no Vue, no Next.js. Rationale:

| Metric | This Stack | React SPA | Next.js SSG |
|--------|-----------|-----------|-------------|
| JS bundle | ~32KB | 140–300KB | 180–350KB |
| TTI | <1.8s | 2.5–4s | 1.5–2.2s |
| LCP | <1.2s | 1.8–3s | <1.5s |
| Complexity | Low | High | Very High |
| Scaling risk | Near-zero | Framework churn | Framework churn |

Vanilla ESM modules with lazy imports hit Lighthouse 95+ consistently because there is no hydration overhead, no virtual DOM reconciliation, and no framework runtime to parse.

### Why 10,000 games doesn't break this

Games are split into **50-game chunks** (~20KB gzipped each). The index file contains only the metadata needed for search and routing (~1KB per game → 10MB uncompressed, ~800KB gzipped). The client only ever loads:

- `games-index.json` once (800KB gzip) — search
- `chunk-000.json` on initial page load (20KB gzip) — first grid
- Subsequent chunks on scroll — 20KB each, loaded 300px before needed

Memory: at 100 loaded games, memory footprint is ~2MB. At 1000 games loaded: ~20MB. JS heap stays under 50MB even with aggressive browsing.

## 2. Performance Strategy

### LCP < 1.8s
- Critical CSS inlined in `<head>` (skeleton layout renders before CSS file loads)
- Fonts loaded via `media="print" onload` trick — zero render blocking
- First game thumbnails (`chunk-000.json`) loaded immediately on boot
- Cloudflare edge: static assets served <20ms from 300+ PoPs

### TTI < 2.5s
- `app.js` is an ES module (deferred by spec)
- `controller.js` is dynamically imported only when game page loads
- AdSense/GTM scripts are `async` — never block main thread
- No polyfills (target: modern browsers only, 2022+)

### Layout Shift (CLS near-zero)
- Game cards have fixed `aspect-ratio: 4/3` — no reflow when image loads
- Skeleton loaders maintain grid dimensions before data arrives
- `contain: layout paint` on game cards prevents cascade reflow

### Image Strategy
- All thumbnails: WebP, 240×180px, q82 — ~15KB per image
- `loading="lazy"` + `width/height` attributes prevent CLS
- `onerror` fallback to placeholder prevents broken image layout

### Cloudflare Caching
```
HTML pages:         no-cache (always fresh for SEO)
JS/CSS bundles:     1 year immutable (content-hashed filenames)
JSON chunks:        1hr CDN + 24hr stale-while-revalidate
Thumbnails:         30 days
Sitemap:            24hrs
```

### trendingScore Formula
```
TS = log10(playCount + 1) × 1000
   + e^(-daysSinceAdded / 30) × 500
   + BayesianRating × 200

BayesianRating = (rating × ratingCount) / (ratingCount + 100)
```

Bayesian prior of 100 votes prevents a new game with 2 ratings of 5 from beating an established game with 10,000 ratings of 4.8. New games get a recency boost that decays exponentially over 30 days.

## 5. Monetisation Architecture

### Ad Zones & Placement Rationale

| Zone | Size | Location | Viewability | CPM Range |
|------|------|----------|------------|-----------|
| Leaderboard Top | 728×90 | Above fold, below header | ~85% | $1.50–3.00 |
| Sidebar Sticky | 300×250 | Right sidebar, sticky | ~75% | $2.00–5.00 |
| Between Games | 320×100 | After every 8 cards | ~60% | $1.00–2.50 |
| Preroll Overlay | 640×480 | Before game loads (5s) | ~95% | $5.00–15.00 |
| Game Footer | 728×90 | Below iframe | ~45% | $0.80–1.80 |

**Behavioural psychology:**
- **Preroll** works because user intent is highest at game-start. The 5-second countdown with a visible timer creates acceptance (users understand the value exchange). Skip becomes available at 5s — completion rate ~65%.
- **Sticky sidebar** is visible throughout the session without interrupting gameplay. CPM is 40% higher than standard placements due to dwell time.
- **Between-game ads** at 8-card intervals match natural scroll pauses — users stop to read card titles. Viewability 60%+ vs 20–30% for footer-only.

### RPM Optimisation
1. **Header bidding** (Prebid.js) — run before AdSense fallback. Increases RPM 30–80% vs waterfall.
2. **Lazy-load ads below fold** — improves page speed (LH score), which increases organic traffic, which increases total revenue even if per-impression rate is unchanged.
3. **Category targeting** — horror category allows adult advertisers; sports allows sports betting (UK). Custom ad units per category.
4. **Refresh rate** — sticky sidebar refreshes every 60s of active viewport time (IAB-compliant).

## 7. SEO Strategy

### Rank For Game Titles
The primary SEO target is **`{game name} unblocked`** and **`{game name} online`** — highest volume, lowest competition in this niche.

**On-page signals:**
- `<title>`: `{Game Title} - Play Free Online | HTMLplay`
- `<h1>`: Game title only
- `<meta name="description">`: shortDescription field
- URL: `/game/{slug}` — slug contains game name + "unblocked"
- Schema.org `VideoGame` structured data with `aggregateRating`

**Content depth:** Game description should be 200–500 words minimum. The `description` field in the schema feeds this. Thin content (<100 words) will not rank.

**Internal linking mesh:**
- Every game page links to 6 related games
- Every category page links to top 20 games in that category
- Sidebar links all categories from every page
- Breadcrumbs on all non-home pages

**Sitemaps:** Split by type with priority signals. Google crawls `sitemap-games.xml` in one batch — all 10,000 pages submitted simultaneously on each deploy.

## 8. Smart Controller System

### Classification Tiers
1. **Cache** (0ms) — previous profile stored in localStorage
2. **Metadata** (0ms) — explicit `controls.scheme` in game JSON
3. **Heuristic** (2ms) — keyword scoring across title/category/tags
4. **Fallback** — generic d-pad + 2 action buttons

### Supported Schemes
| Scheme | Left Control | Right Control |
|--------|-------------|---------------|
| `fps` | Joystick (WASD) | Swipe camera + context buttons |
| `platformer` | D-pad (arrows) | Jump/slide buttons |
| `racing` | ◀▶ pair | Accelerate/brake buttons |
| `sports` | Joystick | Button cluster (shoot/pass/sprint) |
| `puzzle` | None | Tap hint only |
| `generic` | D-pad | A/B buttons |

### Cross-Origin Constraint
Most aggregated games are cross-origin iframes. Direct DOM injection fails silently. The controller dispatches synthetic `KeyboardEvent` to `document` (top-level) AND attempts `iframe.contentWindow.postMessage`. Games that listen at `window` level receive the events; deeply sandboxed games may not — this is an inherent limitation of cross-origin iframe security.

## 9. Security

### Iframe Sandbox
```html
<iframe
  sandbox="allow-scripts allow-same-origin allow-popups allow-forms"
  referrerpolicy="strict-origin-when-cross-origin"
  allow="autoplay; fullscreen"
>
```

`allow-scripts` is required for games to function. `allow-same-origin` is required for localStorage-using games. **Never add `allow-top-navigation`** — this allows iframe breakout attacks.

### Content Security Policy
A full CSP that blocks XSS while allowing game iframes from common origins:
```
Content-Security-Policy: default-src 'self'; 
  script-src 'self' 'nonce-{nonce}' https://pagead2.googlesyndication.com https://www.googletagmanager.com;
  frame-src *;
  img-src * data:;
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
```

`frame-src *` is intentional — game hosting URLs are too diverse to enumerate. The iframe sandbox provides the security instead.

### Anti-Hotlinking
Cloudflare WAF rule: block requests where `Referer` is not `htmlplay.io` for game files under `/games/`. This prevents other sites from embedding your game files.

## 10. Deployment

### First Deploy
1. `git clone` → create Cloudflare Pages project linked to repo
2. Set build command: `npm run build`
3. Set output directory: `.` (root — static files)
4. Add secrets to GitHub: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, `CF_ZONE_ID`
5. Push to `main` → auto-deploys

### Adding Games
1. Add game files to `/games/{slug}/`
2. Add game entry JSON to `/games-raw/{slug}.json`
3. Add thumbnail to `/assets/thumbnails-raw/{slug}.jpg`
4. Push to `main` → pipeline auto-generates chunks, index, sitemap

### Rollback
Cloudflare Pages keeps previous 100 deployments. In the Cloudflare dashboard: Pages → htmlplay → Deployments → select any previous build → "Rollback to this deployment" (instant, no rebuild).
